# SafeMe Emergency Broadcast — Μελλοντικό πλάνο

## A. Γιατί χρειάζεται

Σε μια πραγματική κατάσταση ανάγκης, ο χρήστης δεν πρέπει να χρειάζεται να ανοίγει χειροκίνητα το SMS για κάθε έμπιστη επαφή και να πατά αποστολή μία προς μία. Η διαδικασία αυτή μπορεί να είναι αργή, αγχωτική ή πρακτικά αδύνατη όταν ο χρήστης βρίσκεται σε κίνδυνο.

Τα SMS links μέσα από PWA ή browser δεν μπορούν να εγγυηθούν αξιόπιστη αποστολή σε πολλαπλούς παραλήπτες, ειδικά σε iPhone. Η συμπεριφορά του λειτουργικού συστήματος και της εφαρμογής Messages μπορεί να διαφέρει ανά συσκευή, έκδοση iOS ή ρυθμίσεις χρήστη.

Η τρέχουσα ροή διαδοχικών SMS, όπως SMS 1/2 και 2/2, είναι αξιόπιστο fallback για το MVP, επειδή μειώνει την πιθανότητα αποτυχίας λόγω πολλαπλών παραληπτών. Δεν αποτελεί όμως το τελικό σύστημα emergency broadcast που χρειάζεται η εφαρμογή μακροπρόθεσμα.

## B. Τρέχουσα λύση MVP

Στο τρέχον MVP, το SafeMe ανοίγει την εφαρμογή SMS της συσκευής με προσυμπληρωμένο μήνυμα προς μία έμπιστη επαφή κάθε φορά. Ο χρήστης πρέπει να πατήσει χειροκίνητα **Send/Αποστολή** μέσα στην εφαρμογή SMS.

Οι επαφές χειρίζονται μία προς μία για λόγους αξιοπιστίας, ιδιαίτερα σε iPhone/PWA περιβάλλοντα όπου ένα SMS link με πολλούς παραλήπτες δεν είναι αξιόπιστο. Προς το παρόν δεν αποστέλλεται κανένα SMS αυτόματα από το SafeMe ή από backend υπηρεσία.

## C. Μελλοντική λύση

Η μελλοντική λύση είναι ένα backend emergency broadcast σύστημα. Όταν ο χρήστης ενεργοποιεί την ειδοποίηση, ο server θα στέλνει alert σε όλες τις συγχρονισμένες έμπιστες επαφές, χωρίς να απαιτείται από τον χρήστη να ανοίγει χειροκίνητα SMS για κάθε επαφή.

Το SMS composer της συσκευής πρέπει να παραμείνει ως backup επιλογή, ώστε ο χρήστης να έχει εναλλακτικό τρόπο ειδοποίησης αν η backend αποστολή αποτύχει, αν δεν υπάρχει σύνδεση ή αν ο provider δεν είναι διαθέσιμος.

Το backend πρέπει να καταγράφει κάθε προσπάθεια παράδοσης ή αποστολής, να κρατά delivery logs και να εμφανίζει στον χρήστη ποιες επαφές ειδοποιήθηκαν ή για ποιες επαφές δεν επιβεβαιώθηκε παράδοση.

## D. Possible providers

Πιθανοί providers ή κανάλια για μελλοντική αξιολόγηση, χωρίς καμία υλοποίηση σε αυτό το στάδιο:

- Twilio για SMS αποστολές.
- Vonage για SMS αποστολές.
- MessageBird για SMS ή πολυκαναλική αποστολή.
- Email provider για email broadcast ή test mode.
- Push notifications αργότερα, κυρίως για native app έκδοση.

## E. Requirements before implementation

Πριν υλοποιηθεί οποιαδήποτε αυτόματη αποστολή, χρειάζονται τουλάχιστον τα παρακάτω:

- Επαληθευμένος λογαριασμός χρήστη.
- Συγχρονισμένες έμπιστες επαφές στο backend.
- Συναίνεση επαφών ότι μπορούν να λαμβάνουν emergency alerts.
- Support email για επικοινωνία, παράπονα και αναφορές προβλημάτων.
- Privacy policy που εξηγεί τι δεδομένα αποθηκεύονται και πώς χρησιμοποιούνται.
- Terms of use που περιγράφουν περιορισμούς, ευθύνες και αποδεκτή χρήση.
- Test mode ώστε να δοκιμάζεται η ροή χωρίς πραγματικά emergency alerts.
- Abuse prevention για αποτροπή ψεύτικων ή κακόβουλων ειδοποιήσεων.
- Rate limits ανά χρήστη, επαφή, συσκευή και χρονικό διάστημα.
- Cost control για αποφυγή απρόβλεπτου κόστους SMS/provider.
- Delivery logs με σαφή διάκριση μεταξύ προσπάθειας αποστολής, επιτυχούς αποστολής και επιβεβαιωμένης παράδοσης όπου υποστηρίζεται.

## F. Safety and legal notes

Το SafeMe δεν πρέπει να ισχυρίζεται ότι εγγυάται παράδοση μηνύματος ή ότι εγγυάται διάσωση. Κανάλι όπως SMS, email ή push notification μπορεί να αποτύχει λόγω δικτύου, συσκευής, provider, λάθους αριθμού ή άλλων παραγόντων.

Ο χρήστης πρέπει πάντα να καλεί το **112** ή τις τοπικές υπηρεσίες έκτακτης ανάγκης όταν υπάρχει άμεσος κίνδυνος. Το emergency broadcast προς έμπιστες επαφές είναι βοηθητικό εργαλείο και όχι αντικατάσταση των επίσημων υπηρεσιών έκτακτης ανάγκης.

Οι επαφές πρέπει να γνωρίζουν ότι έχουν δηλωθεί ως έμπιστες επαφές και ότι μπορεί να λάβουν emergency alerts. Η εφαρμογή πρέπει να αποθαρρύνει ψεύτικες ειδοποιήσεις, spam και κάθε μορφή κατάχρησης.

## G. Suggested phases

### Phase 1: Manual SMS fallback, current system

Διατήρηση της τρέχουσας ροής manual SMS fallback. Η εφαρμογή ανοίγει SMS composer και ο χρήστης πατά αποστολή χειροκίνητα για κάθε επαφή.

### Phase 2: Backend email broadcast test mode

Προσθήκη backend test mode για email broadcast, ώστε να δοκιμαστεί η λογική ειδοποίησης όλων των επαφών χωρίς κόστος SMS και χωρίς ισχυρισμούς πραγματικής emergency παράδοσης.

### Phase 3: Backend SMS provider in test/sandbox

Ενσωμάτωση SMS provider σε test ή sandbox περιβάλλον. Η φάση αυτή πρέπει να ελέγχει templates, logs, rate limits, error handling και κόστος πριν από οποιαδήποτε παραγωγική αποστολή.

### Phase 4: Production SMS broadcast with cost/rate limits

Ενεργοποίηση παραγωγικού SMS broadcast μόνο αφού υπάρχουν verified accounts, consent, abuse prevention, cost controls, rate limits, delivery logs και ξεκάθαρο safety/legal wording.

### Phase 5: Native app push notifications and background location

Μελλοντική native app έκδοση μπορεί να προσθέσει push notifications και background location, εφόσον επιτρέπεται από τις πλατφόρμες και υπάρχει σαφής συναίνεση χρήστη.

## H. Product wording

Πιθανές μελλοντικές ετικέτες κουμπιών και status messages:

- «Ειδοποίηση όλων»
- «Άνοιγμα SMS backup»
- «Εστάλη σε 2/2 επαφές»
- «Δεν επιβεβαιώθηκε παράδοση»
